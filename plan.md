# 当前计划

## 已完成

### 1. 工作流主干已闭环

当前仓库已经具备从想法到章节写作的基础闭环：

- `workflow/00-global.md`：全局规划
- `workflow/01-expand.md`：故事点展开
- `workflow/02-merge.md`：多线交汇处理
- `workflow/03-chapters.md`：章节映射、交接卡、可写性约束
- `workflow/04-draft.md`：单章正文生成与成稿自检
- `workflow/05-update-state.md`：正文写后状态回填

这意味着，项目已经不再是“只有 Prompt 片段”，而是已经形成了可执行的写作工作流骨架。

### 2. 章节规划层已完成的补强

`workflow/03-chapters.md` 已完成以下关键补强：

- 增加章节可写性证明
- 增加场景级字数预算
- 增加薄场景判废
- 增加高风险章节类型加严检查
- 增加最低可撑体量判断
- 增加单章装载过满提示
- 增加整卷实跑时的批次备注要求

当前 `03` 已经不只是“把故事点映射成章节”，而是开始约束：

- 这一章是否真的写得满
- 是否天然会滑成摘要稿
- 是否把太多章级内容硬塞进同一章

### 3. 正文生成层已完成的补强

`workflow/04-draft.md` 已完成以下关键补强：

- 增加成稿准入闸门
- 增加场景不足失败判定
- 增加连续失败暂停规则
- 增加首章特殊要求
- 增加连续长跑补充检查
- 增加 `3000+` 先润色压缩的体量偏高复查

当前 `04` 已经不只是“写正文”，而是开始明确判断：

- 什么叫正文章节
- 什么叫摘要稿失败
- 什么叫场景不足失败
- 字数偏高时应该先压缩，而不是直接打回

### 4. 状态回填层已完成的补强

`workflow/05-update-state.md` 已完成以下关键补强：

- 增加正文准入检查
- 禁止把摘要稿、提纲稿、复盘稿直接回填成状态
- 增加整卷实跑时的批次级状态账本要求

当前 `05` 已经不再是“写完就记一下”，而是明确要求：

- 只有成稿级正文才能回填
- 回填必须服务下一章和下一批次

### 5. 已完成的模具级验证

围绕卷尾高风险段，已经完成了一轮工作流级验证，当前已得到这些结论：

- 新模具已经能拦住旧式摘要稿
- 新模具已经开始识别“不是摘要、但场景太薄”的正文
- 对高风险人物亮相章、理念交锋章，单靠“能写满”还不够，还要检查是否装载过满
- 对 `3000+` 的正文，更合理的处理不是直接失败，而是先做润色压缩复查

也就是说，当前工作流已经从“能不能跑起来”，推进到了“能不能在高风险章节上稳住质量”。

虽然单章和局部高风险样本已经验证过，但还没有完整证明：

这一整套链条能稳定连续跑通。

### 2. 高风险章型的连续验证还没做完

当前已经验证了部分高风险章型，但还没有完成更连续的验证：

- 高密度人物亮相章
- 理念交锋章

- 兑现点堆叠

### 3. `3000+` 压缩流程还需要更多样本验证

目前已经验证出一个方向：

- `3000+` 可以先做润色压缩
- 不需要直接按失败处理

但还没有足够样本证明：

- 压缩后能否稳定保住结构
- 会不会误删关键兑现点
- 会不会把人物味道也一起压薄

### 4. 连续实跑里的几个关键风险还未完成验证

还没有完成连续批次层面的验证：

- 多线是否会发散
- 同类卖点是否会疲劳
- 状态回填是否真的反哺下一批
- 局部返工后，后续批次是否还能顺接

### 5. 工具化暂时不在当前阶段

以下内容目前仍未开始，也不应提前开始：

- 高并发
- CLI 工具化
- 自动数据库
- 多人协作系统
- 大规模批处理

当前阶段仍然先验证“人工执行版本的工作流是否稳定”。

## 继续执行的计划

### 第一步：继续做高风险章型回归验证

当前最近的执行重点，不是直接进入整卷连续实跑，而是继续验证模具本身。

优先验证：

1. 人物亮相章
2. 理念交锋章
3. 卷尾高风险承接章
4. 多线交汇后的承接章

目标不是多写样品，而是确认：

- `03` 是否能提前识别薄场景和装载过满
- `04` 是否能正确区分摘要失败、场景不足失败、体量偏高复查
- `05` 是否只接受成稿级正文

### 第二步：补足 5 章以内的小批次闭环样本

在高风险章型验证继续推进的同时，补足一段更完整的小批次闭环样本。

最小目标是完成一组：

1. 连续章纲
2. 连续正文
3. 连续状态回填
4. 一次批次复查

这一步的目的，是先证明工作流能跑通“小批次闭环”，而不是一上来就压整卷。

### 第三步：小批次稳定后，再进入第一卷连续实跑

只有在前两步稳定后，才进入第一卷层面的连续推进。

进入整卷实跑时，按批次执行：

1. 先生成当前批 5~10 章章纲
2. 逐章生成正文
3. 逐章状态回填
4. 每 3~5 章做一次批次复查
5. 发现问题后，先修工作流，再继续下一批

原则是：

- 不一次性写死整卷
- 不一边出问题一边硬往后堆
- 每批都允许受前一批结果反向修正

### 第四步：连续实跑中持续观察四类问题

进入连续实跑后，重点持续观察：

1. 同类卖点是否疲劳
2. 多线是否失控
3. 状态回填是否真实支撑下一批
4. 高风险章型是否反复触发相同故障

如果问题反复出现，修订顺序保持：

1. 先修 `04-draft.md`
2. 再修 `05-update-state.md`
3. 如仍不够，再修 `03-chapters.md`

不回到大改顶层工作流，只做针对连续实跑问题的小修。

## 当前判断

当前项目的状态，不是“还在搭骨架”，也不是“已经可以放心整卷长跑”。

更准确地说：

- 工作流骨架已经搭好
- 关键闸门已经补上
- 模具已经能识别一部分坏样品
- 但还需要通过连续小批次和高风险章型回归，证明它能稳定工作

因此，当前最合适的策略是：

- 先继续校稳模具
- 再进入连续长跑
- 暂不推进工具化
