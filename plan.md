# 修仙搞笑长篇网文生成器落地计划

## 0. 当前决策

第一版产品暂时固定为：

> 修仙成长主线 + 情境喜剧 / 角色喜剧的长篇网文生产器。

当前阶段不追求同时支持都市、历史等其他题材，也不优先开发完整的通用插件平台。

这里的“通用”仅指同一套系统可以服务不同的修仙搞笑小说项目，不能把某一本
小说的人物、境界名称、宗门、法宝、长期目标或剧情样例写进执行器、公共 Schema
和公共 Prompt。它不表示当前版本需要支持都市、历史或其他题材。

保留以下未来替换边界：

- 项目设定与工作流分离
- 正文与状态数据分离
- 修仙规则与执行器分离
- 搞笑风格规则与执行器分离
- 模型提供方与执行器分离

这样可以先把一个题材做深、做稳定，同时避免以后扩题材时完全推倒代码。

## 1. 最终产品目标

用户提供一本修仙搞笑小说的项目资料后，系统能够：

1. 使用现有工作流补齐全局设定。
2. 建立修炼体系、资源规则和成长边界。
3. 建立稳定的搞笑引擎、角色反差和禁用笑料。
4. 建立并人工批准双主角成长、终局、分卷和全书粗单元图。
5. 按总纲顺序一次展开一个约 `10～20` 章的故事单元。
6. 每次准备 `3～5` 章可直接写作的章节细纲。
7. 由代码循环调用 API，逐章生成正文。
8. 每章执行长度、剧情、修炼一致性和搞笑质量检查。
9. 只有正文通过后才提交状态变化。
10. 每 `3～5` 章压缩一次账本，控制后续上下文。
11. 失败时停在当前章，修复后从断点继续。
12. 完成整个故事单元后输出评审报告。

面向用户是“一次启动一段 10～20 章的生产任务”，面向系统内部仍以单章作为最小生成、验收和事实提交单位。

## 2. 内容目标

### 2.1 双引擎

每个有效故事单元必须同时有两条驱动力：

```text
修仙成长引擎：目标、阻碍、行动、资源、能力变化、代价、阶段结果
搞笑引擎：人物反差、信息差、规则错位、场面失控、回调或语言节奏
```

搞笑不能替代剧情，修仙成长也不能压得喜剧风格消失。

### 2.2 修仙成长底线

- 境界、功法、法术、法宝、伤势和资源变化必须可追踪。
- 突破必须有前置条件、过程、资源或代价。
- 不允许为了爽点临时提高能力。
- 不允许同一能力在不同章节出现互相矛盾的效果。
- 战斗结果必须服从已确认的能力边界和现场条件。
- 得到新资源后必须记录数量、限制和去向。
- 阶段胜利不能自动解决长期问题。

### 2.3 搞笑质量底线

优先使用：

- 人物性格反差
- 角色之间的信息差
- 严肃规则与意外执行结果的错位
- 一本正经地处理荒谬局面
- 行动造成的连锁误会
- 前文伏笔或笑点回调
- 对话中的节奏、停顿和立场冲突

禁止依赖：

- 全员突然降智
- 反复使用围观震惊
- 大量网络热梗和过期流行语
- 旁白直接解释“这里很好笑”
- 用羞辱弱者或现实群体制造笑料
- 为了搞笑让危险、损失和人物情感全部失效
- 同一个误会拖延太久且没有新变化
- 每个角色使用相同的说话方式

### 2.4 喜剧与严肃性的关系

搞笑可以缓解压力、制造反差和强化人物，但不能删除后果。

正确结构：

```text
角色用出乎意料的方式处理严肃问题
产生有效笑点
问题仍然发生真实变化
角色得到收益或付出代价
状态被永久记录
```

错误结构：

```text
危险出现
所有人插科打诨
危险自动消失
下一章无人记得
```

## 3. 产品边界

第一版负责：

- 单作者本地使用
- 修仙搞笑题材
- Markdown 工作流
- JSON 状态
- 本地 CLI
- API 调用
- 单章正文生成
- 故事单元批量调度
- 自动检查、重试和恢复

第一版暂不负责：

- 其他题材适配
- Web UI
- 数据库
- 多用户协作
- 自动发布
- 分布式任务
- 同时并发写多本小说
- 完全无人审阅的无限续写

## 4. 系统结构

```text
workflow/
  定义修仙搞笑小说的创作步骤和 Prompt

tools/novel_runner/
  负责调度、API、检查、状态机、提交和恢复

runs/<run-id>/
  保存某一本具体小说的设定、正文、状态和日志

test/matrix-runs/<case-id>/
  保存测试输入和测试产物
```

执行器只处理流程，不在代码中写死某一本小说的人物、境界名称、宗门或剧情。

修仙和搞笑规则可以在第一版工作流中固定，但具体小说内容只能进入 `runs/` 或测试运行目录。

## 5. 工作流阶段

### Phase 0：项目设定

输出：

- 项目定位
- 主角与核心角色
- 世界规则
- 修炼体系
- 资源与代价规则
- 喜剧风格说明
- 双主角独立成长路径与前中后期主动目标
- 最终对手、高潮规则、解决方式与代价
- 分卷骨架与全书连续粗故事单元图
- 人工审批状态与内容哈希
- 禁止事项

### Phase 1：故事单元展开

把一个约 `10～20` 章的阶段目标展开为因果连续的故事点。

每个故事点必须说明：

- 修仙主线推进
- 角色行动
- 阻碍与代价
- 主要搞笑机制
- 状态变化
- 与下一个故事点的因果关系

### Phase 2：多线交汇

只在确有多条线时执行。

重点处理：

- 修炼成长线
- 资源或势力线
- 关系线
- 信息差或误会线

搞笑线不能作为独立小品插入，必须参与因果链。

### Phase 3：章节细纲

每次生成 `3～5` 章细纲。

每章必须包含：

- 目标长度
- 本章意图
- 开场状态
- 必做结果
- 禁止结果
- 修仙推进点
- 搞笑机制与兑现点
- 代价或后果
- 场景级长度预算
- 写作交接卡
- 下一章输入

### Phase 4：正文生成

第一版一次 API 调用只生成一章正文。

正文通过检查后才能进入状态回填。

### Phase 5：状态回填与批次账本

只记录正文实际发生的变化。

每章输出状态增量，每 `3～5` 章输出批次账本。

## 6. 生产层级

| 层级 | 默认规模 | 作用 |
| --- | ---: | --- |
| 故事单元 | 10～20 章 | 一个阶段问题的完整起落 |
| 章纲批次 | 3～5 章 | 控制短期因果、节奏和上下文 |
| 正文请求 | 1 章 | 保证长度和质量稳定 |
| 状态提交 | 1 章 | 防止失败污染后续状态 |
| 账本批次 | 3～5 章 | 压缩后续必读信息 |

默认单章目标为中文正文 `2000～3000` 非空白字符，默认首选区间为 `2200～2500`。该范围保留为运行配置，测试时必须记录实际数值。

## 9. 章节状态机

```text
planned
outline_ready
drafting
draft_generated
draft_failed_provider
draft_failed_length
draft_failed_contract
draft_failed_quality
draft_quality_pending
draft_passed
state_extracting
state_failed
state_ready
committing
committed
locked
```

硬规则：

- 上一章未 `committed`，不能生成正式下一章。
- 正文未达到长度下限，不能提取状态。
- 正文缺少必做结果或出现禁止结果，不能提取状态。
- 状态包含正文未发生内容，不能提交。
- 状态事件和正文必须在同一次原子提交中完成。
- 修改已提交章节后，该章之后的快照、账本和未发布产物全部进入待重验状态。

## 10. 检查分级

### Error：必须阻断

- 长度未达到运行下限
- 输出格式无法解析
- 章节编号错误
- 必做结果缺失
- 出现禁止结果
- 修炼体系硬矛盾
- 无代价突破违反已确认规则
- 状态无正文来源
- 状态机越级
- 失败草稿进入正式状态

### Warning：默认返工，可人工放行

- 场景偏薄
- 搞笑机制与最近章节重复
- 喜剧明显削弱严肃后果
- 主要兑现点偏弱
- 章末钩子不具体
- 配角明显工具化
- 同一误会长期不变化

### Advisory：只提供建议

- 局部表达不够自然
- 某段语言节奏偏慢
- 笑点密度与项目设定略有偏差
- 可增加回调但不影响当前剧情

## 11. 长度闸门

默认统计口径：

```text
只统计正文
排除标题、自检、状态表、作者说明和分隔线
去除空白字符后统计字符数
```

默认处理：

| 正文长度 | 结论 | 动作 |
| ---: | --- | --- |
| 0～1799 | 严重不足 | 重写或回查章纲 |
| 1800～1999 | 轻微不足 | 按场景缺口定向扩写 |
| 2000～3000 | 通过 | 进入其他检查 |
| 3001～3500 | 可接受浮动 | 检查重复和冗余 |
| 3501 以上 | 超长复查 | 先压缩，再决定是否放行 |

自动扩写最多两次。扩写不得：

- 推进下一章事件
- 改变本章结局
- 新增未规划能力或资源
- 依靠重复心理、重复笑话和设定解释凑长度

## 12. 搞笑疲劳控制

每章记录：

- 主要搞笑机制
- 使用对象
- 是否完成兑现
- 是否形成可回调内容
- 与前几章是否重复

默认规则：

- 同一主要搞笑机制不连续使用超过 2 章。
- 同一角色不连续承担完全相同的出丑功能。
- 围观、震惊、误会、嘴硬等反应不能连续成为主要推进方式。
- 一个长期笑点每次回调必须增加新信息、新后果或新关系变化。
- 严肃损失发生时，笑点不得取消损失本身。
- 连续 3 章缺少有效喜剧兑现，检查喜剧引擎是否失效。
- 连续 3 章只有笑点而修仙主线没有变化，暂停批次。

## 13. 上下文策略

生成当前章时读取：

- 当前故事单元目标
- 当前章纲
- 当前章场景预算
- 上一章快照
- 最近批次的 `must_read_next`
- 当前活跃修炼限制
- 当前相关资源
- 当前相关角色的知识与误会
- 当前可用和应暂停的笑点

默认不读取全部历史正文。

需要引用历史细节时，根据事实或线程引用按需加载具体章节片段。

## 14. 自动修改权限

执行器允许自动调整尚未生成章节的：

- 场景顺序
- 场景长度预算
- 局部承接方式
- 搞笑机制
- 小范围资源或信息流转

执行器不得自动修改：

- 故事单元最终目标
- 已确认的修炼体系
- 已提交事实
- 主角长期核心动机
- 项目禁区
- 已确认的重要角色结局

需要改变这些内容时暂停任务，输出修改建议，等待人工确认。

## 15. API 调用策略

一次调用只完成一个任务：

- 规划故事单元
- 生成章纲批次
- 生成单章正文
- 定向扩写
- 格式修复
- 状态提取
- 账本压缩
- 阶段评审

每次调用记录：

- 模型和参数
- Prompt 版本
- 输入快照
- 原始响应
- 解析结果
- 检查结果
- 用量和耗时
- 重试原因

## 17. 实施顺序

### 阶段 0：固定工作流

任务：

- 重写 `workflow/README.md`。
- 重写修仙搞笑模块库。
- 重写 `00～05` 的输入输出和硬规则。
- 统一故事单元、章纲批次和账本批次口径。
- 删除旧工作流中的具体测试样例内容。

验收：

- 工作流只面向修仙搞笑，不混用其他题材目标。
- 工作流不包含某一本具体小说的数据。
- `00～05` 可以人工顺序执行。

### 阶段 1：正文解析与长度检查

任务：

- 拆分章节。
- 识别正文边界。
- 统计实际字符数。
- 输出逐章检查结果。
- 未通过时返回非零退出状态。

验收：

- 不足章节能被准确阻断。
- 不能通过模型自报字数绕过检查。

### 阶段 2：最小 JSON 与状态机

先实现：

```text
run.json
chapter-outlines.json
tasks.jsonl
checks.json
```

验收：

- 状态转换只能按表执行。
- 失败章节不能推进指针。
- 重复命令不会重复提交。

### 阶段 3：单章垂直闭环

目标：尽早跑通一章完整链路。

链路：

```text
导入章纲
生成正文
机械检查
契约检查
状态提取
状态来源检查
原子提交
生成快照
```

验收：

- 一章可以从 `outline_ready` 到 `committed`。
- 中途失败不污染正式状态。

### 阶段 4：断点恢复与版本管理

任务：

- 保存输入快照。
- 保存多个草稿版本。
- 实现任务幂等键。
- 实现运行锁。
- 实现 `status` 和 `resume`。
- 实现修改已提交章节后的依赖失效。

验收：

- 中断后从最后安全点恢复。
- 已提交章节不会被无声覆盖。

### 阶段 5：定向修复

任务：

- 轻微不足时按场景扩写。
- 严重不足时重写或回查章纲。
- 修炼矛盾时针对受影响场景修复。
- 搞笑失效时替换机制，不只增加段子。
- 达到重试上限后暂停。

验收：

- 修复不越过本章边界。
- 修复后重新执行全部检查。

### 阶段 6：状态事件与账本

任务：

- 写入 `events.jsonl`。
- 生成章节快照。
- 生成批次账本。
- 压缩 `must_read_next`。
- 记录修炼状态和笑点疲劳。

验收：

- 下一批主要依赖账本和最近快照启动。
- 压缩后事实不丢失、不改变。

### 阶段 7：故事单元批量调度

任务：

- 实现一次启动 `10～20` 章任务。
- 自动按批次准备章纲。
- 内部逐章生成、修复和提交。
- 每批生成账本。
- 失败后暂停并保留恢复点。
- 故事单元结束后评审。

验收：

- 用户只需启动一次任务。
- 任一章失败后不会提交正式后续章节。
- 修复后能从失败章继续。

### 阶段 8：长链路测试

至少验证：

- 开局双承诺
- 修炼一致性
- 突破代价
- 资源连续性
- 搞笑机制变化
- 误会和知识状态
- 真实受挫
- 严肃章节中的喜剧控制
- 10～20 章阶段收束
- 账本压缩
- 字数缩水
- 中断恢复

### 阶段 9：小批正文实验

只有顺序模式稳定后，才允许一次请求生成 `2～3` 章临时草稿。

后续章节仍需逐章验收。前一章失败时，后续草稿进入隔离区，不得提交状态。

## 22. JSON/JSONL 记忆存储与长链路测试计划

### 22.1 已确定的存储决策

第一版继续使用文件存储，不引入 SQLite、PostgreSQL 或其他数据库。

原因：

- 当前产品是单作者、单机、单进程顺序生成，目标上限不超过 200 章，尚不需要数据库并发能力。
- JSON/JSONL 更便于人工检查、Git 测试归档、错误复现、Schema 调整和状态重建。
- 当前最主要的风险是模型输出错误、上下文膨胀和产物文件过多，数据库本身不能解决这些问题。
- 在状态结构仍处于真实测试校准阶段时引入数据库，会把模型、工作流和存储问题混在同一轮修改中。

硬约束：

- 不允许为每个状态事件、人物、资源、模型调用或重试创建独立数据库或独立 JSON 文件。
- 状态历史使用少量固定 JSONL 文件；当前状态使用固定 JSON 文件原子替换。
- 不永久保存每章完整状态快照，只保留当前快照和故事单元检查点。
- 具体小说数据只能存在于 `runs/` 或脱敏后的 `test/matrix-runs/`，不得进入公共代码、公共 Schema 默认值和公共 Prompt。
- 预留 `Storage` 接口，但第一版只实现 `FileStorage`；数据库实现只有在文件方案确实无法满足需求时才评估。

### 22.2 顶层设计与记忆层级

记忆按作用域和有效期分成五层：

| 层级 | 典型范围 | 作用 | 是否属于已发生事实 |
| --- | --- | --- | --- |
| 项目层 | 全书 | 世界规则、修炼体系、喜剧风格、终局方向 | 规则是事实，未来剧情不是 |
| 卷级层 | 数十至数百章 | 本卷主要矛盾、阶段目标、成长边界 | 目标不是已发生事实 |
| 故事单元层 | 10～20章 | 阶段问题、受挫、破局、代价和收束 | 计划不是已发生事实 |
| 批次层 | 3～5章 | 当前可写细纲、短期因果和交接 | 计划不是已发生事实 |
| 章节事件层 | 1章 | 正文实际发生的资源、关系、境界和知识变化 | 通过提交后才是事实 |

记忆使用规则：

- 上层负责长期约束和检索索引，下层负责具体行动与事实证据。
- 未来计划与正式事实必须存放在不同文件和不同字段中，任何 Prompt 组装都不得把计划状态描述成已经发生。
- 章节偏离细纲时，记录偏差并调整后续计划；不得为了保持原计划而伪造正文未发生的状态。
- 故事单元结束后，把阶段结果、未解决问题和下一单元前置压缩到单元总结；不把全部章节细节继续传递。
- 顶层规则只能通过显式项目修订改变，不能由正文状态抽取模型隐式覆盖。

- `comedy-bible.json`：角色反差、允许的喜剧机制、禁用笑料、回调笑点、疲劳项和严肃场景上限。
- `initial-state.json`：开篇境界、能力、伤势、限制、资源、关系和角色知识状态。
- `master-plan.json`：目标章节、双主角成长路径、阶段主动目标、终局、分卷、全书粗故事单元图和人工审批哈希。

项目层不得包含公共测试样例或其他小说的数据；所有人物、势力、境界名称和剧情目标均属于当前 `run`。

#### C. 故事单元与章节计划

保存到 `planning/`：

- `story-units.json`：从已批准 `master-plan.json` 逐个展开的 10～20章故事单元，ID 和章节范围不得偏离总纲。
- `story-units.md`：便于人工阅读的同源输出。
- `chapter-outlines.json`：当前已规划章节的机器细纲。
- `chapter-outlines.md`：对应的人类可读版本。

每章细纲至少保存：

- 章节号和所属故事单元
- 目标长度及场景预算
- 开场状态
- 章节意图
- 必须发生和禁止发生的结果
- 修仙推进点
- 喜剧机制和兑现点
- 代价或后果
- 章末状态与钩子
- 下一章输入
- 可写性判断和风险

#### D. 正式正文与正文身份

每章永久保留：

- 通过评审的 `draft.final.md`
- 章节号
- 正文非空白字符数
- 正文 SHA-256
- 正式版本号
- 对应章纲版本
- 提交事件 ID

正文草稿、Prompt、原始评审和状态原始输出属于调试产物，不全部永久散落保存，按 22.7 的保留策略处理。

#### E. 正式状态事件

`state/events.jsonl` 是正式状态历史的事实来源，只追加，不整体覆盖。每条事件至少保存：

- `schema_version`
- 全局单调递增 `sequence`
- 稳定且幂等的 `event_id`
- `run_id`、章节号和故事单元 ID
- 来源正文相对路径、版本号和 SHA-256
- 人物与实体变化
- 关系变化
- 境界、能力、伤势和限制变化
- 资源获得、消耗、冻结、解冻和结果余额
- 角色知识、怀疑、误信和淘汰关系
- 伏笔、约束和未解决问题变化
- 喜剧回调、疲劳和暂停使用项
- 下一章输入
- 章纲偏差
- 每项事实的正文连续原句证据

事件写入规则：

- 相同 `event_id` 重复提交必须幂等。
- 事件顺序、章节号和正文哈希必须一致。
- 资源余额、境界前后状态、伤势生命周期和知识淘汰必须通过机械连续性校验。
- 引用不存在的状态 ID、资源 ID 或知识事实 ID 时，不接受事件，进入受限重试。
- 失败状态响应不得进入正式事件文件。

#### F. 当前状态快照

`state/current.json` 是提高读取速度的缓存，不是独立事实来源。保存：

- 快照对应的最后正式章节、最后事件序号和事件哈希
- 当前活动人物与实体
- 当前关系
- 当前境界、能力、伤势和限制
- 当前非零资源余额
- 当前有效知识、怀疑和误信
- 当前活动伏笔、约束和未解决问题
- 当前可回调笑点、疲劳项和暂停使用项

快照不得复制已经归档且对后续无作用的历史细节。快照损坏或与事件不一致时，必须能从 `initial-state.json + events.jsonl` 重建。

#### G. 故事单元检查点

`state/checkpoints.jsonl` 每完成一个故事单元追加一条检查点，最多约每 10～20 章一条。保存：

- 故事单元 ID 和章节范围
- 对应最后事件序号和哈希
- 压缩后的阶段状态
- 已解决问题
- 未解决问题
- 下一单元入口状态
- 修仙成长与资源净变化
- 关系净变化
- 知识差净变化
- 可继续和应暂停的喜剧机制

检查点用于从较近位置重建历史状态，但不能替代正式事件。

#### H. 批次账本

`ledgers/current.json` 保存当前有效账本，`ledgers/history.jsonl` 保存历史批次账本。每份账本保存：

- 批次与章节范围
- `must_read_next`，上限 8 条
- 当前修仙进展
- 当前资源
- 当前关系
- 当前知识差
- 当前伏笔和约束
- 喜剧回调与疲劳风险
- 可归档内容
- 暂停使用项
- 下一批调整

已解决误信、耗尽资源和解除伤势不得继续作为活动项进入下一批账本。

#### I. 模型调用与运行日志

固定使用少量 JSONL 文件：

- `logs/tasks.jsonl`：任务开始、输出保存、接受、失败和恢复。
- `logs/api-calls.jsonl`：任务、模型、Token、耗时、状态和错误类型。
- `logs/runtime-events.jsonl`：运行暂停、恢复、提交、修订、归档和迁移。

日志不得保存 API Key、认证头或完整环境变量。网关错误正文需要经过部署信息脱敏后才能进入公开测试归档。

### 22.4 不应保存或不应长期保留的信息

- 真实 API Key、认证头和完整 `.env` 内容。
- 模型隐藏推理或无法验证的内部解释。
- 把未来大纲复制到当前事实快照中的内容。
- 已解决状态的无限历史副本。
- 每章一个永久完整快照。
- 每个事件、每次重试或每次调用一个独立 JSON 文件。
- 失败且未通过校验的模型状态作为正式事实。
- 相同 Prompt 和响应的多份无差别副本。
- 可由正文、事件或汇总报告重新生成的中间文件。

### 22.6 文件式原子提交与恢复

每章正式提交顺序：

1. 正文通过机械检查和独立质量评审。
2. 状态抽取输出通过 JSON、证据、ID 和连续性校验。
3. 写入固定的 `state/transaction.json`，记录旧指针、目标章节、事件 ID、正文哈希和预期文件哈希。
4. 把事件作为一行追加到 `state/events.jsonl`，刷新并同步到磁盘。
5. 根据正式指针范围内的事件构建新快照，写入 `current.json.tmp`，校验后原子替换 `current.json`。
6. 原子更新 `run.json`，最后正式提交章节作为提交边界。
7. 写入运行事件并清空或删除 `transaction.json`。

恢复规则：

- 正式事件范围以 `run.json.last_committed_chapter` 为边界；超出指针的尾部事件不得直接视为正式事实。
- 发现事务文件时，根据正文哈希、事件 ID、事件序号和正式指针决定完成事务或回退到旧指针。
- 发现 JSONL 最后一行截断时，只允许修复未正式提交的尾行；中间行损坏必须硬失败。
- 快照落后或领先于正式事件时，从最近合法检查点或初始状态重建。
- 事件重复但哈希相同则幂等复用；相同 ID 对应不同内容时硬失败。
- 恢复操作必须记录在 `runtime-events.jsonl`，不得静默修改历史。

### 22.7 调试产物与文件数量控制

保留级别：

- 当前章节：保留全部 Prompt、草稿、评审、状态原始响应和重试版本。
- 当前故事单元：保留详细调试产物，便于定位连续性问题。
- 已完成故事单元：把调试产物压缩为一个 `artifacts/unit-xxxx.zip`，长期散落文件删除或迁入归档。
- 已通过并归档的旧测试：只保留固定测试证据；原始 API 日志和部署错误正文不提交 Git。
- 自动化测试：所有临时目录必须使用自动清理机制，即使测试失败也不得残留海量文件。

文件数量硬门槛：

- 单个运行的结构化记忆长期文件不超过 15 个，不含最终正文和故事单元调试归档。
- 完成 200 章并执行单元归档后，单个运行长期文件总数目标不超过 300 个。
- 任一单目录不得出现超过 1,000 个文件。
- 不允许文件数量随状态事件数量线性增长；100,000 条事件仍应主要保存在固定 JSONL 文件中。
- 自动化测试完成后，测试临时根目录必须为空或被删除。
- 文件数量超过门槛时立即停止真实模型长测，先定位产物生命周期问题。

### 22.8 上下文检索与 Prompt 组装

正文 Prompt 默认只读取：

1. 项目层必要硬规则。
2. 当前卷或阶段目标。
3. 当前故事单元目标和必须保留问题。
4. 最近一份批次账本，`must_read_next` 不超过 8 条。
5. 与当前章纲实体、资源、伏笔和知识差直接相关的当前状态子集。
6. 最近一章的明确交接状态。
7. 当前章节细纲和场景预算。

默认排除：

- 全部历史正文。
- 已确认且当前章节不相关的知识。
- 已耗尽资源、已解除伤势和已归档伏笔。
- 未来章节的具体结果。
- 其他小说的任何数据。
- 调度状态、更新时间等不影响创作的运行元数据。

检索测试必须证明：

- 当前章需要的活动状态全部出现。
- 已解决或无关信息不进入 Prompt。
- 未来计划不会伪装成已发生事实。
- 第二批能主要依赖上一批账本顺畅启动。
- 随章节增加，正文 Prompt 长度不会按完整历史线性增长。

### 22.9 自动化测试分层

#### 第一层：Schema 与文件读写单元测试

- 所有 JSON 文件和 JSONL 行均通过对应 Schema 或结构校验。
- `schema_version` 缺失、未知或不兼容时明确失败。
- 原子文本和 JSON 替换不会产生半文件。
- JSONL 追加、刷新、尾行读取和事件去重正确。
- 安全 run-id 和路径边界继续阻止目录越界。
- 环境变量加载不会写回运行文件和日志。

#### 第二层：状态连续性测试

- 境界突破具有正确前态、后态、前置、成本和新限制。
- 普通状态事件不得擅自改变境界。
- 资源获得、消耗、冻结、解冻和余额顺序计算正确。
- 多种伤势、能力和限制可分别创建、加重、部分恢复和解除。
- 关系变化不会覆盖其他关系维度。
- 角色知识按角色隔离，旧误信可被真实活动事实淘汰。
- 引用不存在或已淘汰的 `supersedes_fact_ids` 时进入受限重试，不污染事件和快照。
- 状态证据必须是最终正文中的连续原句。

#### 第三层：事务与故障注入测试

在以下每个位置模拟进程中断：

- 写事务文件前后
- 追加事件前后
- 写临时快照期间
- 原子替换快照前后
- 更新正式章节指针前后
- 清理事务文件前后

逐项验证：

- 正式章节指针不会跳跃。
- 同一事件不会重复生效。
- 未提交状态不会成为正式事实。
- 恢复后快照等于从事件重建的结果。
- 中断后不产生无限临时文件或恢复副本。

#### 第四层：记忆检索测试

- 从顶层设计逐级选择当前章节需要的信息。
- 计划记忆和事实记忆严格区分。
- 已解决误信、伤势、资源和伏笔不会继续进入活动上下文。
- 多角色同一事实的不同认知不会混淆。
- 账本缺少当前境界、非零资源或未解决知识差时硬失败。
- `must_read_next` 超过 8 条时拒绝正式账本。
- 相同主要喜剧机制连续超过 2 章时标记疲劳风险。

#### 第五层：文件数量与存储压力测试

不调用外部模型，机械模拟：

- 200 章顺序提交。
- 至少 100,000 条状态事件。
- 10～20 个故事单元检查点。
- 反复启动、关闭和恢复。
- 100 次不同提交阶段的故障注入。
- 已完成单元的调试产物压缩与清理。

验收：

- 结构化记忆文件数量不超过 15。
- 200 章归档后长期文件总数不超过 300。
- 不出现按事件数量增长的独立文件。
- 临时测试目录全部清理。
- 两次完整状态重建结果完全幂等。
- 当前快照正常加载时不需要扫描全部历史正文。

#### 第六层：离线工作流回归

- 保留 10～12 章离线长链路回归。
- 覆盖动态章纲、正文检查、质量评审、状态提交、账本和单元收口。
- 覆盖第一批完成后只依赖压缩账本启动第二批。
- 覆盖章节失败、恢复、修订失效和从中间章节重建。
- 离线夹具不得通过重复文本冒充真实小说质量，只验证执行器链路和机械闸门。

#### 第七层：真实模型 10 章确认

按现有 T14、T15、T16 顺序执行：

- T14 先修复并完成，作为模型切换和恢复诊断轮。
- T15 使用全新小说实例，从第 1 章开始固定同一模型路由和 Prompt 参数。
- T15 已证明最终产物安全但效率门槛失败；T16 使用长度安全区间和压缩 Prompt 做独立校准。
- T16 门槛继续采用下一阶段计划中的固定指标，不因结果临时改变。
- 每次调用记录模型、Token、耗时、失败类型和是否产生正文返工。
- 内容评审使用现有评分表，总分至少 12 分且无硬失败。

#### 第八层：40 章连续测试

只有 T15 全部门槛通过后启动：

- 4 个连续的约 10 章故事单元。
- 每个单元结束执行状态重建、账本检查、质量评审和继续/停止决定。
- 至少覆盖一次跨单元伏笔回收、一次旧知识淘汰、一次伤势或限制生命周期、一次带成本成长和一次关系长期变化。
- 第 4 单元平均正文 Prompt 字符数不得相对第 1 单元失控增长；记录每单元平均值和增长来源。
- 喜剧机制疲劳、角色声音趋同和严肃后果被取消均作为停止条件。
- 任一状态污染或文件数量越界立即停止，不继续消耗模型预算。

#### 第九层：100～200 章长期测试

只有 40 章通过后才允许启动：

- 先运行约 100 章，按故事单元逐段放行，不一次性无条件跑到底。
- 100 章通过后才决定是否扩到最多 200 章。
- 每 20 章检查文件数量、上下文长度、每章 Token、状态事件体积、重建时间和喜剧疲劳。
- 至少执行一次中段章节修订，验证后续事件、检查点、账本和未发布产物失效与重建。
- 长测结束必须能够从初始状态和正式事件重建最终快照，并与运行快照一致。

### 22.10 指标与停止条件

每轮真实测试至少记录：

- 正式提交章节数
- 最终字数合格率
- 首稿长度合格率
- 正文返工章节数
- 每章场景预算偏差
- 总调用、成功、失败和无效重试
- 总 Token、每章 Token、平均输入上下文
- 状态格式错误数
- 资源、境界、伤势、关系和知识连续性错误数
- 快照重建差异数
- 账本必读项最大值
- 喜剧机制最长连续重复章数
- 文件总数、结构化记忆文件数和临时残留数
- 人工修订和断点恢复次数

立即停止条件：

- 错误状态进入正式事件或正式快照。
- 正式章节指针跳跃或重复提交造成双重状态变化。
- API Key 或认证信息进入运行日志、测试归档或 Git 候选文件。
- 文件数量超过本计划硬门槛。
- 连续两章无法通过正文质量硬门槛。
- 上下文或 Token 随章节明显线性增长且账本压缩无法控制。
- 同类结构失败在两个独立小说测试中重复出现。

### 22.11 实施顺序

1. T14 与 T15 已完成完整评审、重建和脱敏归档；保留原始测试证据，不破坏性迁移。
2. 在当前存储实现上完成 T16 前的长度安全区间、Prompt 投影和评审自动重试校准。
3. 执行独立 T16 真实 10 章测试；只有固定效率与质量门槛全部通过才允许进入存储改造。
4. 在不调用模型的前提下实现 `FileStorage v2`：固定 JSON/JSONL、当前快照、单元检查点和调试产物生命周期。
5. 增加旧布局兼容读取和只读迁移审计；不破坏 T12～T16 原始运行。
6. 完成 Schema、连续性、故障注入、检索和 200 章文件数量压测。
7. 使用已经归档的真实正文事件进行离线重放，证明 v1 与 v2 最终快照一致。
8. 全部离线门槛通过后，使用 `FileStorage v2` 启动 40 章真实测试。
9. 40 章通过后再逐步扩到 100 章，最后决定是否进行最多 200 章测试。
10. 只有出现多进程写入、多人 Web 服务、百万级复杂查询或 JSON 方案经压测确实无法满足需求时，才重新评估数据库。

### 22.12 第一版完成标准

第一版可以视为完成，必须同时满足：

- 修仙成长 + 情境/角色喜剧工作流不绑定某一本小说。
- 用户可以一次启动一个 10～20 章故事单元，系统内部逐章安全提交。
- JSON/JSONL 记忆在 200 章机械压测下不会产生海量文件。
- 正文、计划、事实状态和调试产物边界明确。
- 任一中断位置都能恢复，正式事件和章节指针不会污染。
- 当前状态可从初始状态与正式事件完全重建。
- 40 章真实连续测试通过；100～200 章测试根据前一阶段结果逐步放行。
- 真实 Key、活动小说运行和原始供应商日志不进入 GitHub。
- 所有公共代码、Schema、Prompt 和文档只固定修仙搞笑题材规则，不固定具体小说内容。
