# 决策

## 结论

T17 的十章正式产物、自动质量评审、连续性账本和状态重建全部通过，但未同时满足固定效率门槛。本轮结论为：**自动质量通过，T17 不放行；暂不进入 FileStorage v2、存储压测或约 40 章连续测试。**

自动评审中的“通过”只表示当前十章正式产物满足章节质量契约，不代表 T17 的额外效率门槛全部通过。

## 通过证据

- 10/10 章正式提交，最终正文共 26,835 个非空白字符，最终长度合格率 100%。
- 首稿长度合格 10/10，合格率 100%，高于至少 80% 的门槛。
- 正文返工章节为 0，低于不超过 2 章的门槛。
- 摘要化、修炼一致性、严肃后果、资源连续性、知识状态、角色声音、多线因果和当前状态硬失败均为 0。
- 第 3 章发生真实受挫并消耗一枚重置印；`left-eye-mirror-burn` 经历加重、受控恢复和正式解除，随后转化为新的操作限制。
- `cen-leaked-exam-route` 误信被新证据正式更正；灵石、清神符、重置印和镜砂余额机械连续。
- 第 9～10 章完成从炼气四层到炼气五层的突破，具备前置、消耗及“回声分裂、暂停高强度监考”等新限制。
- 三份账本的 `must_read_next` 均为 8 条，没有超过上限。
- 喜剧机制相邻重复为 0，同机制最长连续为 1 章。
- 两次 `rebuild-state` 均重建 10 章快照，`changed_chapters` 为空，状态重建幂等。
- API 调用 48 次，恰好满足不超过 48 次的门槛；同时低于 60 次安全上限。
- 三次断点恢复均从安全边界继续，没有污染正式章节、状态事件或账本。

## 未通过的固定门槛

- 总 Token 为 621,200，高于不超过 550,000 的目标 71,200 Token；但仍低于 750,000 的安全上限。
- 平均输入上下文为 20,152.92 字符，高于不超过 20,000 字符的目标 152.92 字符。

因此不能把本轮解释为“接近通过”后直接扩大规模，也不临时放宽 T17 指标。

## 重试与修复验证

- T16 的根因是证据重试提示只要求“逐字引用”，却没有给模型可直接选择的正文段落边界；模型容易删除、改写或拼接证据文字，最终被正确的逐字证据校验拒绝。
- 修复后，评审与状态重试提示会附带编号的逐字段落目录；首轮提示、公共接口和证据校验规则均未改变。
- T17 中第 1 章和第 9 章的状态提取发生内容重试，最终都在既定预算内通过，未人工修改正文或状态结果。这提供了真实模型状态重试路径的正向证据。
- 本轮没有自然触发评审逐字证据内容重试，因此评审重试路径的直接证据仍来自自动化回归测试；不能把 T17 表述为真实模型下完整覆盖了所有重试分支。
- 四次失败调用均为远端传输问题：第 1 章状态提取 TLS EOF、第 7 章状态提取 TLS EOF、第三批规划的 TLS EOF/HTTP 524、以及第 10 章评审的远端关闭/TLS EOF。前三次进程级中断经安全恢复完成，第三批规划失败由既定重试处理。

## 效率分析

成功调用的 Token 分布如下：

- 状态提取：13 次，225,059 Token。
- 正文生成：10 次，142,888 Token。
- 分章规划：7 次，96,145 Token。
- 正文评审：10 次，90,385 Token。
- 账本生成：3 次，58,010 Token。
- 故事单元规划：1 次，8,713 Token。

状态提取是最大消耗项，占总 Token 的约 36%。三次额外成功的状态重试扩大了重试提示上下文，使总 Token 超过放行门槛；平均上下文也因此轻微越线。

## 下一步

1. 暂停 FileStorage v2 和长篇扩容测试。
2. 只针对状态提取及其证据重试上下文做最小压缩，保留逐字证据校验、状态白名单、资源连续性和段落目录能力。
3. 为压缩行为补充正常、异常和边界回归测试，并重新运行小说工作流全量测试、编译与差异检查。
4. 使用全新故事执行一次最终独立十章校准 T18；门槛保持不变。只有 T18 同时通过质量和效率门槛，才进入 FileStorage v2。
