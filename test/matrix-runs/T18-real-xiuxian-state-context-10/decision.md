# 决策

## 结论

T18 的十章正式产物、自动质量评审、连续性账本、平均上下文门槛和状态重建全部通过，但总 Token 未满足固定效率门槛。本轮结论为：**自动质量通过，T18 不放行；不进入 FileStorage v2、存储压测或约 40 章连续测试。**

自动评审中的“通过”只表示十章正式产物满足章节质量契约，不代表 T18 的质量与效率双门槛全部通过。

## 通过证据

- 10/10 章正式提交，最终正文共 25,609 个非空白字符，最终长度合格率 100%。
- 首稿长度合格 10/10，合格率 100%，高于至少 80% 的门槛。
- 正文返工章节为 0，低于不超过 2 章的门槛。
- 摘要化、修炼一致性、严肃后果、资源连续性、知识状态、角色声音、多线因果和当前状态硬失败均为 0。
- 第 3 章发生真实受挫并消耗一枚 `storm-reset-pin`；`right-hand-thunder-numbness` 经历加重、部分恢复和正式解除，解除后仍保留独自值守限制。
- `pei-leaked-strike-window` 误信被新证据正式淘汰；下品灵石、定雷符、重置钉和导雷铜条余额机械连续。
- 第 9～10 章完成从炼气五层到炼气六层的突破，具备前置、消耗和 `post-breakthrough-thunder-echo` 新限制。
- 三份账本的 `must_read_next` 均为 8 条，没有超过上限。
- 喜剧机制相邻重复为 0，同机制最长连续为 1 章。
- 两次 `rebuild-state` 均重建 10 章快照，`changed_chapters` 为空，状态重建幂等。
- API 调用 48 次，恰好满足不超过 48 次的门槛，同时低于 60 次安全上限。
- 平均输入上下文为 18,550.79 字符，低于不超过 20,000 字符的门槛。T17 的上下文超标项在本轮已消除。
- 三次断点恢复均从安全边界继续，没有重复提交正文、污染正式状态或丢失第十章。

## 未通过的固定门槛

- 总 Token 为 576,775，高于不超过 550,000 的目标 26,775 Token；但仍低于 750,000 的安全上限。

因此 T18 不能放行，也不能因为调用数和平均上下文已经通过而临时放宽 Token 门槛。

## 重试、恢复与长等待诊断

- 四次失败调用均为远端传输问题：第 3、5、8 章状态提取发生读超时，第三批合并规划的两条模型路由均收到 HTTP 524。
- 第三批规划失败后，既定流程自动把第 8～10 章拆成单章规划；第十章规划最终由备用 `gpt-5.6-sol` 完成，单次耗时 826.688 秒。
- 规划与正文路由配置的 600 秒、状态路由配置的 300 秒通过 `urlopen(..., timeout=...)` 执行。该值约束阻塞读而非整笔请求的绝对墙钟时间，上游持续或间歇传输时，实际等待可以超过配置秒数。这解释了运行期间看似无期限等待的现象。
- 第 8 章状态首次有效响应错误地使用 `injury + resolve` 解除伤势，契约要求使用 `recovery`；v2 定向修复后通过。
- 第 10 章状态首次有效响应再次恢复已在第 8 章解除的 `right-hand-thunder-numbness`；连续性校验正确拒绝，v2 删除错误变化后通过。
- 第 2、8、10 章发生状态内容重试；第 3、5、8 章发生状态传输恢复。所有恢复均保持原 Prompt、路由、预算和验收门槛不变。

## 效率分析

成功调用的 Token 分布如下：

- 状态提取：13 次，189,900 Token。
- 正文生成：10 次，140,494 Token。
- 分章规划：7 次，95,673 Token。
- 正文评审：10 次，86,544 Token。
- 账本生成：3 次，55,367 Token。
- 故事单元规划：1 次，8,797 Token。

与 T17 相比，总 Token 从 621,200 降至 576,775，减少 44,425；平均上下文从 20,152.92 降至 18,550.79，减少 1,602.13。状态提示压缩有效，但三次状态内容重试仍使状态提取成为最大 Token 项，且四次传输失败把调用数推到 48 次上限，最终没有效率余量。

## 下一步

1. 暂停 FileStorage v2、存储压测和约 40 章连续测试。
2. 在修改代码前，单独提出并确认两项最小改动：为模型调用增加绝对墙钟截止时间；继续压缩状态内容重试上下文，但保留活动状态白名单、资源连续性和逐字证据目录。
3. 为绝对超时、状态解除后重复恢复、状态重试上下文补充正常、异常和边界回归测试。
4. 回归测试通过后，用全新故事执行 T19 独立十章校准；质量与效率同时通过后，才进入 FileStorage v2。
