# Phase 0：修仙搞笑项目全局设定

> 目的：把用户创意整理为可持续写作的项目设定、修炼规则、喜剧圣经和经人工批准的全书总纲。

## Step 0：输入盘点

### 最低输入

- 主角或核心视角角色
- 基本修仙创意
- 开局处境
- 希望的搞笑方向
- 目标篇幅范围

### 可选输入

- 结局方向
- 世界观
- 修炼体系
- 金手指或特殊能力
- 核心配角
- 长期关系
- 主要势力
- 长期对手
- 平台和语言风格
- 禁止剧情
- 禁止笑料

### 输入盘点 Prompt

```text
你是一位修仙搞笑长篇网文策划师。

请先盘点用户输入，不要直接开始扩写剧情。

用户输入：
{{用户输入}}

输出：

1. 用户已经明确提供的信息
2. 仍然缺失的信息
3. 可以安全补全的项目
4. 必须让用户确认后才能确定的重大项目
5. 每项信息来源：用户明确 / 模型建议 / 待确认

不得覆盖用户已经确认的设定。
不得预设具体境界名称、宗门、法宝或结局。
```

通过条件：

- 用户输入与模型补全明确区分。
- 重大设定没有被静默决定。
- 开始下一步所需信息齐全或已列出候选。

## Phase 0-A：故事承诺与终点方向

本阶段回答：这本书长期让读者期待什么，主角最终要走到哪里。

### 必须确定

- 主角当前最缺少什么
- 主角长期想得到什么
- 修仙成长的主要吸引点
- 搞笑风格的主要吸引点
- 主角不能轻易放弃的核心关系或原则
- 长期问题的最终收束方向
- 结局应保留的情感或关系结果

### Prompt

```text
你是一位长篇修仙搞笑网文结构师。

输入：
{{输入盘点结果}}

任务：

如果用户尚未给出终点方向，提出 3 个可持续的候选方向。
如果用户已经给出终点，提供 2 个结构变化方案和 1 个保守校验方案。

每个方案必须包含：

- 一句话故事承诺
- 主角长期目标
- 修仙成长终点
- 核心关系终点
- 搞笑风格如何持续到中后期
- 最终需要解决的核心问题
- 不应提前解决的内容
- 主要风险

不要写具体章节。
不要用空泛的“成为最强”代替人物目标。
```

## Phase 0-B：修炼体系与成长边界

### 必须确定

1. 境界或成长层级
2. 每个层级的能力边界
3. 突破条件
4. 突破代价
5. 资源种类和稀缺程度
6. 功法、法术和法宝的基本规则
7. 伤势与恢复规则
8. 越级行为的允许条件
9. 全书成长上限
10. 已确认例外

### 原则

- 上限确定后不能无说明增加更高层级。
- 能力提升必须带来新问题或新责任，不只增加数字。
- 低级资源不能在后期无限失效，必要时说明替代关系。
- 突破不能仅靠偶然笑料完成。
- 金手指必须有边界、成本或使用条件。
- 同一能力的效果必须保持一致。

### Prompt

```text
你是一位修仙体系设计与连续性编辑。

输入：

项目资料：
{{项目资料}}

用户已有修炼设定：
{{已有设定，没有则留空}}

任务：

在继承用户设定的前提下，输出：

1. 境界或成长层级表
2. 每层能力边界
3. 大致进度范围
4. 突破条件与代价
5. 资源体系
6. 功法、法术、法宝规则
7. 伤势与恢复规则
8. 越级条件
9. 金手指边界
10. 禁止清单
11. 来源标注

搞笑事故可以影响突破过程，但不能绕过既定条件。
```

## Phase 0-C：喜剧圣经

### 必须确定

- 整体喜剧密度：轻喜 / 中等 / 高密度
- 主角的喜剧角色定位
- 核心配角各自的反差来源
- 主要喜剧机制
- 可以回调的长期笑点
- 不允许使用的笑料
- 严肃场景的喜剧上限
- 语言风格和现代词汇边界

### 角色喜剧定位

每个核心角色至少记录：

- 认真目标
- 最在意的体面或原则
- 容易失控的条件
- 与其他角色的反差
- 说话方式
- 不能被破坏的人物底线

### Prompt

```text
你是一位情境喜剧设计师，同时理解修仙长篇的连续性。

输入：

项目资料：
{{项目资料}}

核心角色：
{{角色资料}}

用户希望的搞笑方向：
{{搞笑方向}}

任务：

建立本项目的喜剧圣经：

1. 喜剧密度
2. 主角的喜剧定位
3. 核心配角的反差和语言习惯
4. 允许使用的主要喜剧机制
5. 可长期回调的内容
6. 禁止笑料
7. 严肃场景的喜剧上限
8. 搞笑疲劳预警
9. 前 20 章的机制轮换建议

必须保证：

- 角色不是为了段子临时变笨
- 搞笑结果会影响关系、信息或行动
- 不依赖网络梗堆砌
- 不取消修仙世界的真实危险
```

## Phase 0-D：全书总纲与人工审批

Phase 0 不能只确定“当前十几章写什么”。必须先建立全书目标范围、双主角成长路径、终局冲突、分卷骨架和连续的粗故事单元图。这里不生成逐章细纲，但要保证后续每个 10～20 章单元都有固定位置，不能由执行器临时续算。

正式数据写入 `runs/<run-id>/config/master-plan.json`。

### Phase 0-D1：双主角成长路径与主动目标

- 为两位主角分别确定独立修炼路线、前中后期里程碑和终局状态。
- 按前期、中期、后期划分主动目标阶段；每个阶段必须同时写明两位主角各自目标与共同目标。
- 主角稳定 ID 应与 `config/initial-state.json` 一致。
- 不能用“被事件推着走”代替主动目标。

### Phase 0-D2：终局冲突闭合

- 明确最终对手、对手目标和双方不可调和的核心矛盾。
- 明确高潮成立所需规则，防止临近结局临时追加能力。
- 明确终局解决方式与不可逆代价。
- 终局不等于提前写死所有过程，但必须能反向约束中前期铺垫。

### Phase 0-D3：分卷骨架

每卷必须包含：

- 稳定卷 ID、标题和连续章节范围
- 本卷阶段目标与卷末高潮
- 资源获得/消耗计划
- 喜剧机制计划
- 关系变化
- 至少一项不可逆选择

各卷章节范围必须连续、无重叠并完整覆盖全书目标范围。

### Phase 0-D4：全书粗故事单元图

在每卷内部安排连续的 10～20 章粗故事单元。每个单元使用不会变化的稳定 `unit_id`，并包含：

- 单元标题
- 确切章节范围
- 阶段目标
- 主要阻碍
- 至少一次真实受挫或代价
- 阶段兑现
- 退出状态
- 不得提前解决的长期问题
- 与下一单元的因果接口

粗单元只规定“这一段必须完成什么”，不在 Phase 0 铺逐章场景。所有粗单元必须连续、无重叠、完整覆盖所属卷，单元长度只能为 10～20 章。

### Phase 0-D5：人工审批

总纲先保持 `approval.status=draft`。完成自检并由用户确认后，依次执行：

```powershell
novel validate-master-plan --run <run-id>
novel approve-master-plan --run <run-id>
```

审批会记录不含审批元数据的内容哈希。批准后修改成长路径、终局、卷或粗单元，哈希会失效；重新批准前禁止规划故事单元、导入计划、生成章纲、运行故事单元或直接写正文。

### Prompt

```text
你是一位修仙搞笑长篇网文总纲师。

输入：

故事承诺与终点：
{{Phase 0-A}}

修炼体系：
{{Phase 0-B}}

喜剧圣经：
{{Phase 0-C}}

目标篇幅：
{{目标篇幅}}

任务：

规划可反向约束正文的全书总纲。

要求：

1. 先确定双主角独立成长路径和前中后期主动目标。
2. 再确定最终对手、高潮规则、解决方式与代价。
3. 分卷范围必须连续覆盖目标章节。
4. 每卷必须规划资源、喜剧、关系变化和不可逆选择。
5. 每个粗单元约 10～20 章，章节范围固定且连续。
6. 每个粗单元至少有一次代价、一次可见兑现和一个后续接口。
7. 不把搞笑线独立成与主线无关的小品。
8. 标注用户提供、模型补全和待确认内容。

输出 `master-plan.json` 所需结构，不生成逐章细纲或正文，不自行批准。
```
