# Phase 0：全局设定

> **入口定位**：兼容完整输入和不完整输入。
>
> **核心原则**：
> - 用户给了，就直接使用
> - 用户没给，就由模型补全
> - 补全部分必须标注来源
> - 进入 Phase 1 之前，先给用户审核
>
> **包含四个步骤**：
> - **输入盘点**：区分用户已给 / 缺失待补 / 待确认
> - **0-A 终点推演**：从起点推导终点，或校验用户已给终点
> - **0-B 体系补全**：锁定战力/层级/禁区，防止后期崩坏
> - **0-C 顶层框架**：产出待审核的主线冲突、阶段麻烦、顶层故事点树

---

- 阶段性麻烦
- 是否需要多线
- 模块/节拍结构建议
- 顶层故事点树
- 章节预算建议

## Phase 0-B：战力天花板

> **适用原则**：
> - 修仙/玄幻/高武等题材：强烈建议执行
> - 非战力题材：可把“战力体系”降级为“层级体系 / 权力体系 / 信息体系”
> - 如果用户已给体系，就继承并校验；如果没给，就由模型补全

### 战力体系约束规则

请在开始故事规划之前，锁定以下参数：

#### 1. 境界天花板
```
最高境界：________（如：大乘期 / 真仙 / 祖境）
总境界数：________（如：10个境界）
主角最终达到：________（≤ 最高境界）
```

**铁律**：
- 最高境界一旦设定，**全书不出现更高境界**。没有"飞升后发现上面还有更高的世界"
- 如果确实需要扩展世界，只能**横向扩展**（同级境界的不同体系、不同种族、不同路线），不能纵向扩展
- 主角的最终境界 ≤ 最高境界。你可以不到顶，但不能超顶

#### 2. 进度映射
```
全书共 ____ 章
主角境界推进路线：
  第1章：____（起始境界）
  约第____章：____
  约第____章：____
  ...
  约第____章：____（最终境界）
```

**铁律**：
- 每个境界的章节跨度必须明确。不能出现"主角卡在金丹期300章然后突然连续突破5个境界"
- 境界越高，章节跨度越大（后期突破一章就升一级 = 战力崩了）
- 全书最后20%篇幅内不应有新境界突破（留空间给情感收束和结局）

#### 3. 战力锚点
定义3-5个"战力参照物"，作为全书战力的固定标尺：
```
锚点1：____（如：苏家老祖 = 元婴期，第1章就存在）
锚点2：____（如：天剑宗宗主 = 化神期，第200章登场）
锚点3：____（如：圣地之主 = 合体期，第500章登场）
...
```

**作用**：读者通过参照物校准主角的实力。如果第50章主角能秒杀一个"化神期"的对手，但第200章化神期的宗主还是大Boss → 崩了。

#### 4. 战力代价
每次重大突破必须有代价，防止"闭关三天连升三级"：
```
突破类型        代价示例
小境界突破      时间+资源（正常）
大境界突破      时间+资源+特殊机缘+失败风险
越级战斗        燃烧寿命/透支潜力/外力反噬
开挂/奇遇       欠下因果（未来必须偿还）
```

#### 5. 禁止清单
- [ ] 禁止"飞升后发现上面还有更大世界"（除非全书一开始就声明有多层世界）
- [ ] 禁止"这个Boss其实只是XX的棋子"使用超过2次
- [ ] 禁止主角在全书最后10%篇幅内获得超过1个新境界
- [ ] 禁止"上古大能"的实力描述超过已设定的天花板
- [ ] 禁止"今天突破，明天就同级无敌"——给每个境界留适应期

## Phase 0-C：顶层故事点树

> 完成 0-A 和 0-B 后，运行此步骤。
>
> **目标**：先产出一份**待审核结构方案**，而不是默认直接开写。

### Prompt 正文

```
你是一位资深网文结构师。你的任务是根据我提供的用户输入、模型补全结果、战力/层级体系约束，以及叙事模块库，构建一份【待审核的顶层结构方案】。

## 模块库参考

{{在此粘贴 workflow/modules.md 的完整内容}}

## 核心原则

1. **故事点驱动，而非章节驱动**：拆分的最小单位是"故事点"——一个完整的叙事节拍。章节数量是后续映射的结果。
2. **模块驱动拆分**：每个需要展开的故事点，必须从模块库中选择一个模块。
3. **多线并行**：多条叙事线独立展开，在模块4（多线并拢）处交汇。
4. **战力约束**：故事点的推进必须符合已设定的战力进度映射。
5. **压必有争，压不连三**：任何"压"之后必须有"争"；同一模块内不出现连续3个同类型节拍。

## 我的设定

### 用户已明确提供
[粘贴输入盘点中 `user` 字段]

### 模型补全 / 校验后的关键项
- 终点：
- 主线冲突：
- 是否多线：
- 结构建议：
- 主要阶段性麻烦：

### 战力/层级体系
[从 Phase 0-B 确定的体系摘要]

### 目标总章数
[用户给的范围，或补全后的建议范围]

## 输出要求

### 1. 主线与阶段麻烦
先输出：
- 主线核心冲突
- 前期主要麻烦
- 中期升级麻烦
- 后期终局麻烦
- 是否需要多线，如需要，列出各线功能

### 2. 顶层故事点树

用缩进树形结构。节点格式：
`SP-{id} [{模块名}] [{节拍类型}] {名称} — {一句话描述} (预估占比: X%)`

示例：
```
SP-0 [模块4] [根] 全书 — 从废材到破壁者 (100%)
├── SP-1 [模块4] [压] 荒诞开局 — 穿越变父子，两线引入 (20%)
│   ├── SP-1.1 [模块4] [压(A)] 苏尘线：废材+多了个儿子 (8%)
│   │   ├── SP-1.1.1 [叶] [压] 穿越醒来被全族嘲笑
│   │   ├── SP-1.1.2 [叶] [争] 发现婴儿不对——眼神不像婴儿
│   │   ├── SP-1.1.3 [叶] [破] 用现代梗确认身份：基友也穿了
│   │   └── SP-1.1.4 [叶] [得] 父子秘密联盟达成
│   ├── SP-1.2 [模块1] [压(B)] 苏念尘线：婴儿身躯+天才灵魂 (7%)
│   ├── SP-1.3 [叶] [争(交汇)] 族会逼宫——父子第一次联手应对 (3%)
│   └── SP-1.4 [叶] [破] 三月族比之约 + 苏念尘天灵根初显 (2%)
├── SP-2 [模块2] [压] 夹缝求生 — 螺旋施压 (18%)
...
```

### 3. 模块选择理由
对每个使用了模块的节点，简要说明。若某一段不适合固定套用现有模块，也要明说，并给出临时结构建议。

### 4. 交汇点标注
标注所有交汇点：涉及的线、交汇方式、交汇驱动事件。

### 5. 体系进度检查
每个顶层故事点标注主角大致境界范围，确保与战力进度映射一致。

### 6. 来源标注
标出：
- 哪些部分直接继承用户输入
- 哪些部分由模型补全
- 哪些部分必须由用户审核拍板

### 7. 关键约束检查
- [ ] 模块2之后是否都接了模块3？
- [ ] 是否出现连续3个同类型节拍？
- [ ] "压"之后是否都有对应的"争"？
- [ ] 预估占比总和 = 100%？
- [ ] 战力进度是否与设定的进度映射一致？
```
