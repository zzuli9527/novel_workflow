# Phase 3：故事点 → 批次章节细纲

> 目的：从当前故事单元中取连续故事点，生成 3～5 章可直接写正文的章节细纲。

## 输入

```text
项目定位
修炼体系
喜剧圣经
当前故事单元
已展开故事点
多线交汇结果（如有）
上一章状态快照（如有）
上一批账本（如有）
运行长度策略
```

## Prompt

```text

不要写正文。
不要提前解决故事单元后段问题。
不要为了凑章数拆出天然写不满的章节。
```

## 1. 章节拆分原则

章节数量由事件密度决定，不机械均分。

一章应当包含：

- 一个主要行动目标
- 一个具体阻碍或条件变化
- 至少一个可见结果
- 修仙、资源、关系或信息的一项真实变化
- 一个主要喜剧机制或明确的低喜剧说明
- 一个可承接的章末动作

如果一个故事点只能稳定支持半章，应与相邻内容合并。

如果一个故事点包含多个独立目标、多个重大转折或多个不能压缩的场景，应拆章。

## 2. 场景最低要求

默认每章至少 2 个有效场景，特殊短章需说明理由。

有效场景必须至少包含：

- 场景目标
- 参与角色
- 具体阻碍
- 行动与回应
- 状态变化
- 进入下一场景的原因

只有说明、回忆、气氛或重复对话的片段不能单独算有效场景。

## 3. 场景级长度预算

每个场景记录目标长度。

预算应包含：

- 开场动作和场面建立
- 对话或对抗回合
- 修仙规则、资源或能力的现场体现
- 喜剧触发与反应
- 结果和后果
- 场景过渡

场景预算总和应落在章节目标范围附近。

如果只有依靠以下内容才能达到目标长度，章纲不合格：

- 重复心理
- 重复笑话
- 大段设定解释
- 多次复述同一判断
- 无功能环境描写
- 旁白总结人物和主题

## 4. 修仙推进要求

每章必须标注：

- 本章涉及的修炼规则
- 本章能力边界
- 资源获取或消耗
- 伤势或限制
- 是否允许突破
- 本章结束后的修炼状态

没有修为变化的章节也可以成立，但必须推动资源、认知、关系、任务或世界规则。

重大突破章必须在章纲中提前写明：

- 前置条件
- 使用资源
- 失败风险
- 突破过程
- 新能力边界
- 代价或后遗症

## 5. 搞笑设计要求

每章记录：

- 主要喜剧机制
- 由哪些角色触发
- 笑点如何参与因果
- 喜剧兑现点
- 是否使用前文回调
- 本章后是否需要暂停该机制

高压、受伤、离别等严肃章节可以降低笑点数量，但仍要保持角色语言和行为风格一致。

禁止：

- 为了制造笑点让角色忽略明显危险。
- 依赖一句网络梗作为本章主要喜剧兑现。
- 每章都以群体震惊、沉默或打脸收尾。
- 同一角色连续两章承担完全相同的出丑功能。
- 误会仅靠不说一句正常话维持。

## 6. 每章兑现

每章必须有至少一个明确兑现：

- 目标取得阶段结果
- 获得有效信息
- 资源发生变化
- 关系发生变化
- 能力理解发生变化
- 喜剧误会产生真实后果
- 前文承诺得到局部回收

连续两章不能只有铺垫。

## 7. 首章要求

如果这是项目第 1 章，尽快完成：

- 主角是谁
- 主角当前处境
- 修仙世界的一个关键规则
- 眼前必须处理的问题
- 主角与普通人的行为或判断差异
- 本书主要喜剧气质
- 一个具体行动钩子

禁止把首章写成：

- 世界观说明书
- 境界表介绍
- 没有行动的日常闲聊
- 只靠网络梗吸引人
- 只有搞笑，没有长期问题

## 8. 批次节奏

当前 3～5 章批次至少包含：

- 一次有效推进
- 一次阻碍、代价或条件变化
- 一次明确兑现
- 一次喜剧机制切换或升级
- 一个批次结束后的新行动方向

如果连续三章修仙主线没有变化，批次不通过。

如果连续三章没有有效喜剧兑现，检查项目喜剧引擎是否没有进入场景。

## 9. 写作交接卡

每章必须输出交接卡：

```text
开场地点与时间
开场角色状态
已知事实
角色之间的信息差
当前修炼状态和限制
当前资源
本章必须发生
本章禁止发生
本章主要喜剧机制
本章必须保留的严肃后果
章末状态
下一章最小输入
```

## 10. 可写性证明

每章必须回答：

1. 至少有哪些有效场景？
2. 每个场景为什么不能被一句话概括掉？
3. 场景预算如何支撑目标长度？
4. 修仙主线发生了什么真实变化？
5. 喜剧如何参与因果？
6. 本章兑现是什么？
7. 本章结束后哪些状态必须更新？

不能回答时，不进入正文。

## 11. 状态 ID 边界

细纲只规定语义结果，例如“苏叶不再相信姜守改写口令签”，不负责构造最终状态事件。

- 不得在细纲中编造或强制指定 `supersedes_fact_ids`、`state_id`、`resource_id`。
- 只有上一正式快照或最近账本中已经存在的稳定 ID 才可以被复用。
- 本章出现的新证据 ID 不能冒充需要淘汰的旧 `fact_id`。
- 具体状态事件必须等正文通过后，由状态回填阶段根据活动 ID 白名单生成。
- 如果故事只要求纠正旧误信，细纲写清角色、旧认知和纠正结果即可，不写机器实现细节。

## 硬失败

- 场景预算无法自然达到目标长度。
- 章节只有笑点，没有主线变化。
- 章节只有修炼说明，没有场景行动。
- 突破条件与全局体系冲突。
- 资源无来源或无去向。
- 角色为维持误会拒绝正常交流，且没有合理动机。
- 连续两章没有兑现。
- 章末钩子只有模糊感觉，没有具体行动方向。

## 批量处理建议

- 一次只生成 3～5 章细纲。
- 当前批次未完成前，不详细生成远端章节。
- 批次正文完成后，先读取状态和账本，再微调下一批。
- 微调不得改变故事单元最终目标和已确认修炼规则。

## 机器导入产物

人工可读章纲通过后，还必须同步输出一个独立 JSON 文件，供执行器校验和导入。不要让执行器直接猜测 Markdown 标题和列表的含义。

最低外层结构：

```json
{
  "schema_version": "1.0",
  "story_units": [],
  "chapter_outlines": []
}
```

规则：

- `story_units` 使用 `plan.md` 的 `story-units.json` 字段。
- `chapter_outlines` 使用 `plan.md` 的 `chapter-outlines.json` 字段。
- 一个故事单元的 10～20 章可以分 3～5 章逐批策划。
- 可以只导入 `story_units` 并把 `chapter_outlines` 留空；`run-unit` 会在每批开始前调用章纲规划，并让后续批次读取上一批账本。
- 也可以提前导入全部章纲；不接受只覆盖半个标准批次的残缺章纲。
- 每章 `status` 写为 `outline_ready`；故事单元 `status` 写为 `planned`。
- JSON 只能写已经在人工章纲中确认的内容，不得额外补剧情。
- 执行 `novel import-plan --run <run-id> --file <plan.json>`；校验失败时回到本阶段修正，不能手工绕过。
- 单独生成标准批次可执行 `novel plan-batch --run <run-id> --unit <unit-id> --range <start-end>`。
