# Phase 5：正文状态增量与批次账本

> 使用方式：每章正文通过 `04-draft.md` 全部硬闸门后执行。状态回填只提取正文事实，不续写、不补剧情。

## 0. 准入检查

必须同时满足：

- 输入是完整正文章节，不是摘要或提纲。
- 机械长度闸门通过。
- 本章必做结果通过。
- 本章没有修炼体系硬矛盾。
- 本章正文版本已确定。

不满足时输出：

```text
禁止回填，先返工正文。
失败原因：{长度 / 摘要 / 契约 / 修炼矛盾 / 其他}
```

不得继续生成正式状态。

## 输入

```text
本章细纲
本章写作交接卡
本章 final 正文
本章检查结果
上一章状态快照
当前修炼体系
当前喜剧圣经
```

## Prompt

```text

你的任务是从已经通过验收的本章正文中提取状态增量。

你不能续写。
不能把章纲计划当成事实。
不能推测下一章会发生什么。
每个变化都必须能在正文中找到来源。

机器结构化输出中的每个正式变化必须包含 `source_evidence`，其值必须是 final 正文中的连续原句，不得使用概括或改写句代替。
```

## 1. 实体状态变化

只记录本章结束后发生变化的实体：

- 位置
- 身体与伤势
- 情绪和短期意图
- 身份或公开处境
- 当前行动
- 持有物

没有变化的实体不重复抄写。

## 2. 关系变化

记录有行为依据的关系变化：

- 信任增加或下降
- 合作形成或终止
- 新义务
- 新冲突
- 立场变化
- 公开关系与真实关系的差异

不能仅凭一次内心感想判断关系已经永久改变。

## 3. 修炼状态变化

记录：

- 当前境界或层级
- 本章修炼进度
- 新理解
- 新能力及边界
- 功法、法术和法宝变化
- 伤势和限制
- 突破条件完成情况
- 已付出的代价

如果正文没有明确发生突破，不得根据章纲填写突破完成。

机器状态要求：

- 每项使用稳定 `subject_id` 和 `kind`。
- `kind=breakthrough` 时必须同时提供 `from_stage`、`to_stage`、非空 `prerequisites`、`costs` 和 `new_limits`。
- 非突破事件不得改变境界名称。
- 修炼主体必须已经存在于初始状态或上一章快照。

## 4. 资源变化

每项资源记录：

- 名称或引用
- 获得 / 消耗 / 转移 / 损坏
- 数量或程度
- 来源或去向
- 使用限制

禁止只写“资源增加”而不说明是什么。

机器状态要求：

- 使用稳定 `owner_id`、`resource_id` 和 `unit`。
- 操作限定为 `gain / consume / transfer_in / transfer_out / damage / restore`。
- 必须填写正数 `amount`、`source_or_destination` 和 `resulting_balance`。
- 执行器会根据上一余额机械计算；余额为负或声明余额不一致时，状态事件不得提交。

## 5. 知识、秘密与误会

按角色分别记录：

- 已经知道
- 仍然不知道
- 错误相信
- 正在怀疑或调查
- 故意隐瞒
- 公开信息与真实信息的差异

误会必须按角色知识状态记录，不能只写作者视角总结。

机器状态要求：

- 使用稳定 `character_id` 和 `fact_id`。
- 状态限定为 `knows / believes_false / suspects / investigating / conceals`。
- `belief` 必须写该角色当前实际相信的内容。
- 同一章同一角色对同一事实只能有一个结束状态。
- `supersedes_fact_ids` 只能引用同一角色在上一正式状态中仍然活动的旧 `fact_id`。本章新增证据、
  章纲中的技术标签或正文临时措辞不能作为待淘汰 ID；没有合法旧事实时必须使用空数组。

## 6. 势力与世界规则变化

记录本章实际改变的：

- 势力立场
- 组织规则
- 身份权限
- 任务与义务
- 世界规则的新信息

没有变化时可以为空。

## 7. 叙事线程

分为：

- 新增线程
- 本章推进
- 已解决
- 暂停
- 必须在后续继续

每条线程记录最近来源章节。

## 8. 喜剧状态

记录：

- 本章主要喜剧机制
- 已兑现笑点
- 新增可回调内容
- 本章回调了什么
- 回调是否升级
- 哪些机制开始疲劳
- 哪些角色不应继续承担同类笑点
- 严肃后果是否仍然存在

喜剧状态不是收集段子，而是防止长期重复和人物变形。

## 9. 下一章最小输入

只整理下一章必须知道的内容：

- 当前地点与时间
- 核心角色状态
- 当前修炼状态和限制
- 当前关键资源
- 当前最紧迫问题
- 角色之间的重要信息差
- 必须承接的线程
- 可使用或应暂停的喜剧机制

## 10. 偏差判断

判断正文与章纲：

- `none`：无实质偏差
- `minor`：局部执行变化，不影响故事单元目标
- `major`：改变章末状态、修炼体系或后续因果

重大偏差时：

```text
暂停进入下一章
说明是返工正文还是重做后续章纲
```

## 11. 批次账本

每完成 3～5 章，额外生成一次批次账本。

### 必须包含

- 下一批必读，不超过 8 条
- 当前修炼状态和硬限制
- 当前关键资源
- 当前关系和义务
- 当前知识差与误会
- 活跃线程
- 可回调笑点
- 应暂停的喜剧机制
- 可归档内容
- 下一批调整建议

机器账本必须完整覆盖批次结束快照中的结构化活跃状态：

- 每个修炼主体的稳定 ID 和当前境界。
- 每个余额大于 0 的资源，其持有者、资源 ID、数量和单位。
- 每个仍处于误信、怀疑、调查或隐瞒状态的角色知识差。

执行器会把账本与批次结束快照逐项比对；遗漏或余额错误时不生成正式账本。

### 输出格式

```text

1.
2.

- 可继续：
- 应升级：

## 8. 可归档
-

- 需要更换的喜剧机制：
- 需要加强的修仙推进：
- 必须兑现的内容：
- 不得提前解决的内容：
```

## 批次账本硬失败

- `下一批必读` 超过 8 条且未压缩。
- 遗漏当前修炼硬限制。
- 遗漏已发生的资源消耗。
- 误会状态没有按角色区分。
- 疲劳笑点仍被列为默认继续使用。
- 把未生成章节的计划写成事实。

## 提交原则

状态增量通过后：

```text
追加状态事件
生成新快照
更新章节状态为 committed
更新运行指针
```

任何一步失败，正式运行指针保持在上一章。
