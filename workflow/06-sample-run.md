# Phase 6：小样例试跑与归档

> **使用方式**：当 `00-05` 已经具备后，用本文件规范一次真实试跑。
> **目的**：不是生成正式作品，而是验证工作流能否顺畅跑通，并把卡点留下来用于修订。

---

## 试跑目标

第一次试跑不要上来就做大体量项目。建议控制在：

- **总量**：8-15 章
- **题材**：单主线优先
- **交汇**：最多 1 个轻交汇点
- **战力**：简单、少层级
- **节拍覆盖**：至少覆盖一轮 `压 / 争 / 破 / 得`

除了规模控制，本阶段还要覆盖两种输入场景：

1. **完整输入测试**：用户给了较多设定，系统应优先继承，不乱覆盖
2. **不完整输入测试**：用户只给最小创意，系统应合理补齐缺项并明确标注来源
3. **章节字数测试**：细纲和正文是否真的围绕 2000~2500 字组织，而不是到正文阶段再硬撑
4. **首章强开局测试**：首章是否足够快地给出主角、现状、冲突和吸引点
5. **番茄风格测试**：是否具备快节奏、短段落、强钩子，而不是拖沓说明文
6. **合规原创测试**：是否存在违规风险、明显 AI 味、或疑似模仿/抄袭问题

## 归档目录建议

每次样例试跑，建议在仓库下建立一个独立目录，例如：

```text
samples/
  sample-001/
    README.md
    00-global.md
    01-expand.md
    02-merge.md
    03-chapters.md
    drafts/
      ch01.md
      ch02.md
      ch03.md
    states/
      ch01-state.md
      ch02-state.md
      ch03-state.md
    review.md
```

说明：

- `00-global.md`：该样例的 Phase 0 结果
- `01-expand.md`：该样例的 Phase 1 结果
- `02-merge.md`：如有交汇，保存交汇细化；没有可省略
- `03-chapters.md`：章节细纲和交接卡
- `drafts/`：正文
- `states/`：每章写后状态回填
- `review.md`：记录本次试跑暴露的问题

---

## 最低归档要求

一次试跑至少要保存：

1. 顶层故事点
2. 3-5 个叶节点
3. 8-15 章章节细纲
4. 前 2-3 章正文
5. 前 2-3 章状态回填
6. 一份问题复盘
7. 一份输入来源记录（哪些来自用户，哪些来自模型补全）
8. 第一章专项检查（是否抓人、是否非流水账）

如果这些产物没有保存，就不算一次有效试跑。

## 使用原则

1. 试跑的目标是暴露问题，不是把样例写得多漂亮。
2. 不要边跑边大改顶层流程，先把问题记下来。
3. 一次试跑结束后，优先根据 `review.md` 做小范围修订。
4. 重点观察 Phase 0：当用户输入不完整时，系统是否真的做到“缺啥补啥”；当用户输入完整时，系统是否真的做到“不乱覆盖”。
5. 不要只看正文结果，也要回头检查 `03-chapters.md`：如果一章内容天然不够 2000~2500 字，那是拆章问题，不只是写作问题。
