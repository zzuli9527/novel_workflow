# 番茄小说细纲工作流

> 从想法到全书章节细纲，再到逐章写作执行的递归工作流。
>
> **核心思想**：故事点驱动 × 模块化叙事 × 多线并行 × 递归展开 × 战力天花板锁死。

---

## 核心理念

传统做法：1000章 → 均分 10×100 → 再均分 10×10 → 填内容。**问题**：按章节拆分，叙事节奏被机械切割；战力随着地图扩大不断膨胀；"起"太多导致故事跳跃。

本工作流做法：

```
用户想法（只有起点）
    │
    ├─ Phase 0-A: 终点推演 —— 从起点推导3个可能的终点，选最优
    ├─ Phase 0-B: 战力天花板 —— 锁死最高境界、进度映射、战力锚点
    ├─ Phase 0-C: 顶层框架 —— 给定起终点 + 模块库 → 顶层故事点树
    │
    ▼
起点 A ─────── 递归插入故事节点 ───────→ 终点 Z
         │                               │
         每次插入从模块库(1-5)选模块展开
         模块决定节拍序列（压争破得组合）
         展开到"不可再分的故事事件"→ 叶节点
         最后才把叶节点映射到章节
```

- **拆分对象**：故事点（叙事节拍），不是章节
- **拆分方式**：模块驱动递归，不是机械等分
- **章节数量**：故事点映射后的**结果**
- **多线处理**：每条线独立展开，用模块4（多线并拢）交汇
- **战力控制**：天花板在规划阶段锁死，不从纵向突破

---

## 目标调整

这套工作流的目标，不是做高并发平台，也不是先追求复杂基础设施，而是做成一套**单作者可持续执行、可复盘、可返工、能真实写到正文**的工作流。

它应该满足：

- **能跑通**：从一个想法，稳定走到章节细纲和正文
- **能停能续**：中途停下后，下一次能接着写，不靠临场回忆
- **能局部返工**：某一段有问题，优先改叶节点、细纲和章节，不轻易推翻全书
- **能保一致**：人物状态、关系、战力、伏笔有记录，不靠模型“记住”
- **人工可控**：关键节点允许人工确认，不追求全自动

---

## 输入兼容原则

这套工作流**不要求用户一开始就提供完整设定**。真实使用时，输入可能非常少，也可能非常完整。正式原则如下：

1. **用户给了，就直接使用**：不要重复发明用户已经明确给出的设定。
2. **用户没给，就由模型补全**：模型负责补齐终点、主线冲突、阶段性麻烦、多线判断、结构建议等缺项。
3. **补全部分必须显式标注**：让用户知道哪些内容来自自己，哪些来自模型推导。
4. **进入下一阶段前必须审核**：模型补的内容不能默认当真，必须先过用户这一关。

### 输入字段分层

- **必填**：主角、题材、大致思路、目标章数范围
- **选填**：终点、世界观、战力体系、反派、爽点、感情线、多线设想、禁区
- **派生项**：主线冲突、阶段性麻烦、模块选择、顶层故事点树、章节预算

执行时统一遵守：

`先盘点已有输入 -> 再补缺失项 -> 标记来源 -> 交用户审核 -> 再进入后续阶段`

## 落地版原则

为了保证这套流程真的能用于写作，执行时遵守以下原则：

1. **先求单书跑通**：先保证一本书从想法走到正文，不为高并发设计。
2. **每一步有明确输入输出**：Phase 的输出要能直接喂给下一步，不能只留模糊描述。
3. **故事树稳定优先**：顶层树一旦确认，不轻易回炉；返工优先落在叶节点、章节映射和正文。
4. **章节写作按批次推进**：先做一批连续叶节点的细纲，再逐章写，不要全书一次性铺到底。
5. **状态显式记录**：每章写完都要回填状态变化，作为下一章输入。

## 细纲怎么定

这套工作流里的“细纲”，不是先拍一个总章数，再机械均分；也不是为了凑满 2000~2500 字，把一个小动作硬拉成一章。

细纲的制定顺序应当是：

1. **先看叶节点到底承载几个关键动作**
2. **再判断这些动作能自然拆成几章**
3. **再反推每章是否真有足够事件密度支撑 2000~2500 字**
4. **最后才确定章数**

因此，真正决定细纲的，不是“平均分配”，而是下面四件事：

- **事件密度**：一章里是否真的有足够事情发生
- **冲突落点**：本章主冲突多早落地，是否慢启动
- **兑现点**：本章是否给了读者明确获得感，而不是只铺不兑
- **钩子链**：章末是否自然把读者推到下一章

执行时优先级是：

`先保事件密度和追读节奏 -> 再保兑现点和钩子 -> 最后才保章数`

如果一个章节规划出来后，天然只能靠背景说明、重复心理、重复确认、空转对话来撑字数，就说明不是正文不行，而是细纲拆错了，应该优先回到 Phase 3 重新压章、并章或合并叶节点。

## 全局写作约束

这些要求必须前置进入工作流，而不是等到正文阶段再临时补救。

### 1. 章节字数约束

- 默认每章目标字数：**2000~2500字**
- 这意味着：
  - 章节映射阶段就要考虑事件密度
  - 某章如果天然只够写一个很小动作，就不应硬拆成单章
  - 如果一章需要靠重复解释、空转心理、无效环境描写来凑字数，说明前面的拆章有问题

### 2. 番茄向风格约束

这里说的“番茄风格”，不是模仿具体作品，而是遵守番茄优秀网文常见的共性：

- 节奏快
- 信息进入快
- 冲突进入快
- 段落短，阅读阻力小
- 章末必须有继续读下去的理由

严禁：

- 抄袭
- 直接模仿具体作品的桥段、句式、对白
- 用流水账和大段背景说明拖节奏

### 3. 首章高优先级约束

第1章必须特殊对待，不能按普通章节处理。

首章应尽快完成：

- 主角亮相
- 主角现状建立
- 冲突或异常抛出
- 吸引读者继续看的核心钩子落地

禁止首章写成：

- 背景说明书
- 纯设定展示
- 人物自我介绍
- 无冲突的日常流水账

### 4. 正文质量底线

- **去 AI 味**：避免空泛总结、模板化转折、过度解释
- **合法合规**：禁止不良引导，禁止违规内容
- **边界清晰**：热门梗可以玩，但不能靠恶俗羞辱或明显人身攻击制造效果
- **必须原创**：严禁抄袭，严禁套用现成网文桥段和表达

## 最小可落地闭环

真正执行时，不要把这套流程理解为“只生成细纲”。最小闭环应该是：

`想法 -> Phase 0 -> Phase 1 -> Phase 2 -> Phase 3 -> 单章写作 -> 单章复查 -> 状态回填 -> 下一章`

其中：

- **Phase 0-3** 负责把结构做对
- **单章写作** 负责把一章真正写出来
- **单章复查** 负责检查该章是否完成目标、有没有钩子、有没有设定越界
- **状态回填** 负责把本章产生的变化写回工作流记录

只有闭环跑起来，这套方法才算真正落地。

## 使用步骤

### Step 0：阅读模块库

先读 `modules.md`，理解五个模块的节拍序列和适用场景。这是整个工作流的"语法"。

### Step 1：运行 Phase 0

打开 `00-global.md`，按顺序执行三个子步骤：
1. **先盘点输入**：区分哪些是用户明确给的，哪些是缺项
2. **0-A 终点推演**：如果用户没给终点，AI 给 3 个终点方向；如果用户给了终点，只做校验和补充
3. **0-B 体系补全**：如果用户没给战力/层级/禁区，AI 补全；给了就继承
4. **0-C 顶层框架**：综合已有信息和补全信息，产出待审核的主线冲突、阶段麻烦、顶层故事点树

**注意**：Phase 0 的输出不是直接开写，而是先给用户审核。审核通过后再进入 Phase 1。

### Step 2：运行 Phase 1（循环）

对 Phase 0 产出中的**每个非叶节点**：
1. 打开 `01-expand.md`，填入节点信息
2. AI 按模块节拍序列展开
3. 子节点中的非叶节点 → 继续 Phase 1
4. 直到全部为叶节点

建议：**深度优先**，先完整展开一条分支。

### Step 3：运行 Phase 2（按需）

对标记为交汇点的叶节点，打开 `02-merge.md` 细化交叉节奏。

### Step 4：运行 Phase 3

分批处理叶节点：
1. 打开 `03-chapters.md`
2. 每次输入 1-3 个连续叶节点（约 5-20 章）
3. AI 产出逐章细纲
4. 为每章生成写作交接卡
5. 手动维护章号计数器

### Step 5：按章执行正文写作

对已经生成细纲的章节，逐章写作，不建议一口气写太多章。每次只处理 1 章，输入至少包括：

1. **上一章结束状态**：人物、关系、战力、资源、伏笔状态
2. **本章细纲**：核心事件、冲突、信息释放、章末钩子
3. **本章目标**：这一章必须完成什么推进
4. **本章边界**：不能提前揭示什么、不能突破什么、不能跳过什么

正文写完后，立刻做一轮轻量复查：

- 这一章是否完成了细纲目标？
- 章末钩子是否成立？
- 有没有偷跑后续剧情？
- 人物语气和关系是否偏移？

通过后，再进入下一章。

建议直接使用 `04-draft.md` 作为本步骤的标准 Prompt。

### Step 5.5：执行写后状态回填

正文写完并完成本章自检后，不要直接进入下一章，而是立刻回填本章状态。

回填至少包括：

1. **人物状态变化**
2. **关系变化**
3. **战力 / 资源变化**
4. **伏笔变化**
5. **下一章前置状态**

建议直接使用 `05-update-state.md` 作为本步骤的标准 Prompt。

### Step 6：分段复查与局部返工

每写完一个小批次后，做一次分段复查。建议粒度：

- **每 1 章**：轻量复查
- **每 5-10 章**：检查连续性、节奏和钩子链
- **每 15-20 章**：检查人物弧线、战力进度、伏笔回收

如果发现问题，返工顺序建议是：

1. 先改章节正文
2. 再改对应章节细纲
3. 如仍不够，再回改叶节点
4. 只有结构性问题，才回改上层故事点

尽量避免为了解决局部问题，把顶层框架整段推翻。

## 关键约束速查

| 约束 | 说明 |
|------|------|
| ⚠️ 模块2→3 | 螺旋施压后必须接集中收割，否则读者弃书 |
| ⚠️ 压必有争 | 任何"压"之后必须在同一模块内看到"争" |
| ⚠️ 压不连三 | 同一模块内不出现连续3个同类型节拍 |
| ⚠️ 战力天花板 | 全书最高境界一旦设定，不出现更高境界 |
| ⚠️ 故事点≠章节 | 展开时不分配章节号，那是 Phase 3 的事 |
| ⚠️ 每章有钩子 | Phase 3 输出中，每章钩子不为空 |
| ⚠️ 每章写完要回填 | 人物、关系、战力、伏笔变化必须记录 |
| ⚠️ 小改优先 | 先改章节和叶节点，不轻易回改顶层树 |
| ⚠️ 分段复查 | 连续写作后必须做批次复查，避免越写越偏 |
| ⚠️ 每章 2000~2500 字 | 章节规划必须反推事件密度，不能到正文阶段硬撑 |
| ⚠️ 首章必须抓人 | 主角/现状/冲突/吸引点必须尽快到位，禁止流水账开局 |
| ⚠️ 番茄向而非模仿 | 学节奏和信息推进效率，不学具体作品内容 |
| ⚠️ 去 AI 味与合规原创 | 正文不得空泛模板化，不得违规，不得抄袭 |
