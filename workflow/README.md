# 修仙搞笑长篇网文工作流

> 从项目想法到故事单元、章节细纲、正文和状态账本的连续生产工作流。

## 目标

本工作流第一版固定服务于：

```text
修仙成长主线
+
情境喜剧 / 角色喜剧
+
移动端长篇连载节奏
```

它不是单纯的段子生成器，也不是只会升级打脸的模板。

最终要做到：

- 修炼成长有规则、有资源、有代价。
- 搞笑来自人物和事件，不靠全员降智。
- 每章有真实推进，不写成剧情摘要。
- 每章状态可追踪，模型不需要凭记忆续写。
- 一次可以启动约 10～20 章任务，内部逐章验收。
- 任何故事单元展开前，全书总纲已经人工批准且未被改动。
- 中途失败可以停止、返工和恢复。

## 双引擎原则

### 修仙成长引擎

每个故事单元至少推动一项：

- 修为或能力
- 功法、法术或法宝
- 资源获取与消耗
- 伤势、限制或代价
- 身份、关系或势力位置
- 世界规则认知
- 长期问题的解决进度

### 搞笑引擎

优先使用：

- 性格反差
- 信息差和误会
- 严肃规则被意外执行
- 一本正经处理荒谬局面
- 行动引发连锁后果
- 前文回调
- 对话节奏和立场碰撞

搞笑必须改变场面、关系、信息或行动，不能作为与主线无关的小品插入。

## 不可违反的底线

### 修仙底线

- 不临时创造未设定能力解决问题。
- 不无资源、无过程、无代价突破。
- 不让同一境界和能力前后矛盾。
- 不让战斗结果无视能力边界和现场条件。
- 新资源必须记录来源、数量、限制和去向。

### 搞笑底线

- 不让所有角色突然变笨。
- 不连续依赖围观、震惊、嘴硬和打脸。
- 不大量堆网络梗或流行语。
- 不让旁白解释笑点。
- 不让笑点取消真实危险和损失。
- 不通过现实群体羞辱制造效果。
- 不让所有角色说话同一种腔调。

## 输入原则

用户不需要一开始提供完整设定。

### 最低输入

- 主角或核心视角角色
- 基本修仙创意
- 希望的搞笑方向
- 开局处境
- 目标篇幅范围

### 可选输入

- 世界观
- 修炼体系
- 金手指或特殊能力
- 核心关系
- 长期对手
- 结局方向
- 平台与语言风格
- 禁止使用的笑料和剧情

### 处理规则

```text
用户已经给出：直接继承
用户没有给出：模型提出候选
模型补全内容：必须标记来源
重大设定：进入后续阶段前确认
```

## 工作流总览

```text
项目输入
  ↓
Phase 0：全局设定
  项目定位、双主角成长路径、终局、分卷、全书粗单元图、人工审批
  ↓
Phase 1：故事单元展开
  把 10～20 章阶段目标展开为连续故事点
  ↓
Phase 2：多线交汇（按需）
  让成长、资源、关系和信息差通过同一事件相撞
  ↓
Phase 3：章节细纲
  每批生成 3～5 章细纲、场景预算和交接卡
  同时维护一个可由 `novel import-plan` 校验的机器 JSON
  ↓
Phase 4：正文生成
  一次生成 1 章，执行长度和质量检查
  ↓
Phase 5：状态回填
  保存状态增量；每 3～5 章生成压缩账本
```

## 生产层级

| 层级 | 默认规模 | 说明 |
| --- | ---: | --- |
| 全书总纲 | 全书 | 双主角路径、主动目标、终局、分卷和粗单元图 |
| 分卷 | 数十章 | 阶段目标、资源、喜剧、关系与不可逆选择 |
| 故事单元 | 10～20 章 | 一个阶段问题的完整起落 |
| 章纲批次 | 3～5 章 | 短期连续规划 |
| 正文生成 | 1 章 | 最小写作和验收单位 |
| 状态提交 | 1 章 | 最小事实提交单位 |
| 批次账本 | 3～5 章 | 下一批上下文压缩 |

默认单章目标为 2000～3000 个非空白正文字符，默认首选区间为 2200～2500；实际值可以由运行配置覆盖。

## 每章的最低结构

一章至少需要完成：

1. 从上一章确定状态启动。
2. 尽快给出当前问题或行动。
3. 展开至少两个有功能的场景，特殊短章除外。
4. 产生真实的修仙、资源、关系或信息变化。
5. 完成一个明确的阅读兑现。
6. 至少有一个有效喜剧节拍，严肃高压章可以降低密度。
7. 保留真实后果。
8. 给下一章一个具体行动方向。

## 有效喜剧节拍

有效喜剧节拍至少改变一项：

- 角色的判断
- 角色之间的关系
- 当前行动方式
- 信息差
- 场面控制权
- 资源或结果

只插入一句俏皮话，但删掉后完全不影响人物和场面，不算有效喜剧节拍。

## 批次节奏要求

每个 3～5 章批次至少包含：

- 一次明确推进
- 一次阻碍、代价或意外后果
- 一次可见兑现
- 一次喜剧机制变化
- 一个能承接下一批的问题

禁止：

- 连续两章只有准备，没有结果。
- 连续三章只有搞笑，没有修仙主线变化。
- 连续三章完全没有有效喜剧兑现。
- 连续两章使用完全相同的主要笑点结构。

## 正文与状态的关系

```text
细纲是计划
正文是事实来源
状态是已通过正文的结构化结果
账本是状态的上下文压缩
```

状态回填必须以正文为准。细纲中计划发生、但正文没有写出的内容，不得进入正式状态。

## 失败处理

出现以下任一情况，停止当前正式流程：

- 正文低于运行最低长度。
- 正文退化为摘要。
- 必做结果缺失。
- 出现禁止结果。
- 修炼体系硬矛盾。
- 搞笑只能依赖全员降智才能成立。
- 状态记录了正文未发生内容。

失败后：

```text
保存当前草稿
记录失败原因
禁止状态提交
禁止正式生成下一章
定向修复或退回章纲
```

## 文件职责

| 文件 | 职责 |
| --- | --- |
| `00-global.md` | 项目设定、双主角成长、终局、分卷、粗单元图与人工审批 |
| `modules.md` | 修仙推进模块与喜剧机制库 |
| `01-expand.md` | 展开当前故事单元 |
| `02-merge.md` | 处理多线因果交汇 |
| `03-chapters.md` | 生成批次章纲、场景预算和交接卡 |
| `04-draft.md` | 生成单章正文并自检 |
| `05-update-state.md` | 提取状态增量和批次账本 |
| `06-sample-run.md` | 规范测试与归档 |

## 推荐使用顺序

新项目：

```text
00-global
→ 填写 config/master-plan.json
→ validate-master-plan
→ 用户审核并执行 approve-master-plan
→ 从已批准总纲选择下一个固定故事单元
→ 01-expand
→ 必要时 02-merge
→ 03-chapters 生成首批 3～5 章
→ 把完整故事单元章纲写入 import-plan JSON 并执行导入校验
→ 04-draft 写第 1 章
→ 05-update-state
→ 继续下一章
→ 批次结束生成账本
```

续写项目：

```text
读取最近状态快照
读取最近批次账本
确认当前故事单元
从尚未完成的章纲继续
```

## 当前完成目标

第一阶段不是生成整本书，而是稳定完成：

```text
1 个修仙搞笑项目设定
1 份经人工批准的全书总纲
1 个 10～20 章故事单元
2～4 个章纲批次
10～20 章逐章正文
逐章状态增量
批次账本
故事单元评审
```
